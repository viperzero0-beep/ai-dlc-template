# コスト最適化ガイド

AI-DLC は速度と品質を上げますが、AI とCIの使い方次第でコストが膨らみます。
このガイドは **無駄を省きつつ品質を落とさない** ための指針です。

## 1. モデル選択（最重要）

タスクの難易度に応じてモデルを使い分けます。**安価・高速なモデルを既定にし、難所だけ高性能モデルへ切り替える**のが基本です。

| タスク                                                               | 推奨モデルの性格         | 理由                     |
| -------------------------------------------------------------------- | ------------------------ | ------------------------ |
| 要件整理・計画・要約・振り返り（`/inception` `/bolt-plan` `/retro`） | 高速・低コスト           | 推論より構造化が中心     |
| 実装・デバッグ・複雑な設計（`/construction`）                        | 高性能（フラッグシップ） | 正確性と一発合格率が重要 |
| コードレビュー                                                       | バランス型               | 網羅性とコストの両立     |
| ドキュメント・ADR（`/adr`）                                          | 高速・低コスト           | 定型出力が中心           |

> 具体的なモデル名は提供側で更新されます。VS Code のモデルピッカーで現行の「軽量／標準／高性能」を選び、上表の性格に合わせてください。
> GitHub Copilot では高性能モデルが **プレミアムリクエスト**を消費します。難所に集中投下し、単純作業は標準モデルで行いましょう。

## 2. トークン / コンテキストの節約

- **関係する箇所だけ読む**。大量ファイルの一括読み込みや不要な全文検索を避ける。
- **永続コンテキストを活用**する（`docs/ai-dlc/context/`）。同じ説明を毎回 AI に与え直さない。
- 長い会話は要点を残して**新しいセッションに切り替える**（コンテキスト肥大を防ぐ）。
- `copilot-instructions.md` と `.instructions.md` は**簡潔に**保つ（毎回読み込まれるため）。`applyTo: "**"` は避ける。

## 3. CI/CD のコスト

このテンプレートの [ワークフロー](../../.github/workflows/) には以下を組み込み済みです。

- **重複実行のキャンセル**: `concurrency` + `cancel-in-progress`。
- **ドキュメント変更ではビルドをスキップ**: `paths-ignore`。
- **タイムアウト**: 暴走ジョブの課金を防ぐ `timeout-minutes`。
- **最小権限**: `permissions` を絞る。
- **AI レビュー補助は LLM 非依存**: チェックリスト自動投稿はコスト 0。本格的な AI レビューは Copilot code review を利用。

追加のヒント:

- マトリクスビルドは必要最小限に（OS / 言語バージョンを絞る）。
- キャッシュ（依存関係）を活用してジョブ時間を短縮。
- ナイトリービルドや重い E2E は頻度を下げる。

## 4. 運用（Operations）のコスト

- リソースは**適切なサイズ**から始め、オートスケールを使う。
- 不要になった環境・リソースは削除する（特にプレビュー環境）。
- マネージドサービスの選定は**コスト見積りを添えて人間が決定**する。

## コスト・チェックリスト（Bolt / PR ごと）

- [ ] そのタスクに適した最小限のモデルを選んだか
- [ ] 不要なファイル読み込み・検索をしていないか
- [ ] 永続コンテキストを再利用したか
- [ ] CI を無駄に何度も回していないか（古い実行はキャンセルされているか）
- [ ] 新規依存・新規リソースの追加は妥当か（人間が承認したか）
